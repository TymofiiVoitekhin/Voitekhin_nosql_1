Відповіді на питання (Частина 1)

Чому аудіо-характеристики винесені в окремий об’єкт audio_features?
Це логічне групування (Embedding), яке зменшує забруднення кореневого документа та дозволяє витягувати всі характеристики однією проєкцією (audio_features: 1). Таке вкладення вигідне, коли поля логічно пов'язані та зазвичай зчитуються разом. Створює проблеми лише тоді, коли рівень вкладеності стає занадто великим (MongoDB лімітує вкладеність) або якщо потрібно будувати складні індекси на масиві глибоко вкладених об'єктів.

Чому виконавці зберігаються як масив, а не як рядок?
Плоский рядок зі знаком ; унеможливлює швидкий та точний пошук, вимагаючи дорогих Regex-запитів. Масив дозволяє базі використовувати Multikey Indexes. Це кардинально спрощує запити: знайти трек конкретного артиста можна через пряме співставлення find({ artists: "Queen" }), а порахувати аналітику — використавши оператор $unwind.

Що таке $out і чим він відрізняється від $merge?
$out повністю перезаписує вказану цільову колекцію (видаляє стару і створює нову). Ідеально підходить для повних пакетних трансформацій (як у нашому завданні).
$merge дозволяє вставляти результати агрегації в існуючу колекцію: він може оновлювати поточні документи або додавати нові (upsert). Використовується для інкрементальних оновлень даних.

Відповіді на питання (Частина 2)

1. **Для чого використовується інструкція `$unwind`?**
   Оператор `$unwind` застосовується для денормалізації (розгортання) масивів. Якщо документ містить масив із 3-х елементів, `$unwind` замінить цей один документ на 3 окремих клони, кожен з яких міститиме лише один елемент із масиву. У нашому випадку це критично необхідно для агрегації: неможливо згрупувати дані по артистах (`$group`), доки вони "закриті" всередині масиву. `$unwind` перетворює елементи масиву на атомарні значення, по яких база може проводити групування та математичні операції.

2. **Чим `$stdDevPop` відрізняється від `$stdDevSamp`?**
   Ці оператори відповідають за розрахунок стандартного відхилення, але використовуються в різних статистичних сценаріях:
   - **`$stdDevPop` (Population Standard Deviation):** Застосовується, коли у нас є дані по всій генеральній сукупності (Population). У нашому датасеті представлені всі треки для конкретного жанру, які є в базі, тому ми обчислюємо відхилення саме для цього повного набору. 
   - **`$stdDevSamp` (Sample Standard Deviation):** Використовується, коли ми аналізуємо лише невелику вибірку (Sample) з величезного масиву невідомих даних, намагаючись зробити екстраполяцію. У формулі обчислення `Samp` використовується знаменник `N - 1` (поправка Бесселя) для усунення зміщення оцінки. У цій задачі використання `Samp` було б логічною помилкою.

### Відповіді на теоретичні питання

1. **Що буде, якщо змінити поріг кількості треків для виконавців (Запит 1)?**
   - **Якщо знизити до 1:** У топ-10 потраплять виключно "артисти одного хіта" (one-hit wonders) — виконавці, які мають лише один трек у базі, але його популярність становить 99 або 100. Вони повністю витіснять стабільних топ-артистів. Такий рейтинг не несе бізнес-цінності.
   - **Якщо підвищити до >50:** Вибірка кардинально звузиться до гіпер-продуктивних ветеранів сцени. Оскільки утримувати максимальну популярність (90-100 балів) на дистанції у 50+ треків майже неможливо, *абсолютне значення* середньої популярності лідерів цього топу помітно впаде.

2. **Чи зміниться результат, якщо знизити поріг для жанрів з 100 до 50 треків (Запит 3)?**
   - Так, результат може сильно змінитися і стати статистично недостовірним. У статистиці діє Закон великих чисел: чим менша вибірка, тим більша ймовірність випадкових відхилень. Вузькоспеціалізований жанр, що має рівно 51 трек, які випадково виявилися дуже танцювальними, очолить список, обігнавши масові жанри (як pop чи edm, де тисячі треків стабілізують середнє значення). Поріг у 100+ треків відсіює статистичний шум.
